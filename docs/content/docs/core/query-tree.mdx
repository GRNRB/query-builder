A query is plain JSON â€” a **tree** whose root is always a `group`. Each child is either a
`condition` (a leaf) or another `group` (so you can nest `and` / `or` arbitrarily deep). There
are no classes and no methods on the nodes; a query is just a serializable value you can store,
send over the wire, and diff.

```ts
{
  type: "group",
  operator: "and",
  children: [
    { type: "condition", field: "name", operator: "eq", value: "Alice" },
    {
      type: "group",
      operator: "or",
      children: [
        { type: "condition", field: "age", operator: "gt", value: 18 },
      ],
    },
  ],
}
```

## Condition nodes

A `ConditionNode` is a leaf: a single filter as `field` + `operator` + (usually) `value`. The
value's shape depends on the operator's [arity](/docs/core/schema#operators-and-arity).

```ts
{ id: "c1", type: "condition", field: "name", operator: "eq", value: "Alice" }
```

## Group nodes

A `GroupNode` combines other nodes with a logical `operator`. Its `children` are themselves
conditions or groups.

```ts
{ id: "g1", type: "group", operator: "and", children: [ /* ConditionNode | GroupNode */ ] }
```

## Every node has an id

Both node types carry a unique string `id`. Ids are how every operation targets a node â€” to
move, remove, or update it, you pass its id.

## Input variants â€” ids are optional

When you *build* a tree (an initial query, or a node you add), you usually don't invent ids by
hand. The `*Input` types make `id` optional: `ConditionNodeInput`, `GroupNodeInput`, and the
union `QueryNodeInput`. Pass an input, and `normalizeTree` fills in any missing ids.

```ts
// No ids needed â€” normalizeTree assigns them.
{
  type: "group",
  operator: "and",
  children: [{ type: "condition", field: "age", operator: "gt", value: 18 }],
}
```

### normalizeTree

```ts
function normalizeTree(node: QueryNodeInput, generateId?: () => string): QueryNode
```

Recursively assigns an `id` to any node missing one â€” turning an id-less input into a real
query tree. It only fills *missing* ids, so re-normalizing an already-complete tree is safe.
Pass a custom `generateId` to control id generation (defaults to `defaultGenerateId`, i.e.
`crypto.randomUUID()`); see [Custom ids](/docs/guides/custom-ids).

```ts
import { normalizeTree } from "@grnrb/query-builder-core";

const query = normalizeTree({
  type: "group",
  operator: "and",
  children: [{ type: "condition", field: "name", operator: "eq", value: "Alice" }],
});
```

## Immutability

Every operation is **immutable**: it returns a *new* tree rather than mutating the one you
passed. Two properties follow, and both are useful.

**Structural sharing & referential equality.** When something changes, only the nodes on the
path to the change are recreated â€” untouched subtrees keep their references. When nothing
changes, the helper returns the **exact same reference**:

```ts
const next = appendToGroup(tree, "missing-group-id", node);
next === tree; // true â€” no such group, so nothing changed
```

You can detect "did anything change?" with `===`, and React skips re-rendering structurally
shared subtrees. `children` is declared `readonly` as a reminder to transform, not push.

**Safety guards.** A few operations refuse to do something destructive and return the tree
unchanged instead:

- **Removing the root is a no-op** â€” the root anchors the query (dev-mode warning).
- **Moving a node into itself or a descendant is rejected** â€” which would orphan the moved
  subtree (so is moving to a target that doesn't exist).

Because a query is plain immutable data, it's trivial to keep history
([undo/redo](/docs/guides/undo-redo)), [serialize it](/docs/guides/serialization), and pass it
through any store without defensive copying.

<Callout>
Treat a `QueryNode` as frozen. Change it through the [reducer](/docs/core/reducer) or the
[tree helpers](/docs/core/tree-helpers) â€” never by assigning into `node.children` or a
condition's fields directly.
</Callout>
