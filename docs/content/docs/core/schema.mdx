The **schema** is a static description of *what's allowed*: which fields exist, which
operators each field supports, and how groups combine. It carries no state — it's the grammar
your queries are written in — and because it's defined with `defineSchema`, TypeScript infers
its literal types and drives the typing of everything else.

```ts
import { defineSchema } from "@grnrb/query-builder-core";

const schema = defineSchema({
  fields: [
    { id: "name", label: "Name", supportedOperators: ["eq", "contains"] },
    { id: "age", label: "Age", supportedOperators: ["eq", "gt", "lt"] },
  ],
  conditionOperators: [
    { id: "eq", arity: "one", label: "equals" },
    { id: "contains", arity: "one", label: "contains" },
    { id: "gt", arity: "one", label: "greater than" },
    { id: "lt", arity: "one", label: "less than" },
  ],
  groupOperators: [
    { id: "and", arity: "many", label: "AND" },
    { id: "or", arity: "many", label: "OR" },
  ],
});
```

Only `fields` and `conditionOperators` are required; `groupOperators` is optional (but you'll
almost always want it).

## defineSchema

```ts
function defineSchema<const TSchema extends QuerySchema>(schema: TSchema): TSchema
```

An identity function with a `const` type parameter: it returns the schema unchanged but
**preserves its literal types** (powering operator-per-field inference) and, in non-production
builds, **warns** about a duplicate field id, a duplicate condition operator id, or a
`defaultOperator` not in `supportedOperators`.

<Callout title="Always wrap with defineSchema">
Passing a plain object to the reducer or hook works at runtime, but `defineSchema` preserves
the literal types (so operator/field inference works) **and** warns in development about
duplicate ids or a `defaultOperator` that isn't supported.
</Callout>

## Fields

Each field declares an `id`, a human `label`, and the operator ids it `supportedOperators`.
An optional `defaultOperator` (which must be one of `supportedOperators`) is used when a
condition switches to this field.

## Operators and arity

Condition operators and group operators share one shape. The key field is **`arity`** — how
many values an operator takes — which determines the `value` shape of a condition using it.

<AutoTypeTable path="../packages/query-builder-core/src/type.ts" name="OperatorDefinition" />

```ts
type Arity = "none" | "one" | "two" | "many";
```

| Arity    | Example operator       | `value` shape              |
| -------- | ---------------------- | -------------------------- |
| `none`   | `is_set`, `is_empty`   | *(omitted / `undefined`)*  |
| `one`    | `eq`, `contains`, `gt` | a single value `TValue`    |
| `two`    | `between`              | a tuple `[TValue, TValue]` |
| `many`   | `in`, `not_in`         | an array `TValue[]`        |

A `none`-arity operator has **no value at all**:

```ts
// Active field whose only operator is `is_set` (arity: "none") — note: no `value` key.
{ type: "condition", field: "active", operator: "is_set" }
```

This is why a UI typically renders the value input from the operator's arity: `one` shows a
text box, `none` shows *(no value)*.

<Callout title="Value shape is enforced">
Because the schema knows each operator's arity, TypeScript enforces the right `value` per
condition — a `none` operator forbids a `value`, a `one` operator requires one. See
[TypeScript inference](/docs/frameworks/react/api#typescript-inference).
</Callout>

## Group operators

The valid group operators are whatever you put in `groupOperators` (typically `and` / `or`,
with `arity: "many"`). They combine a group's children. The `GroupOperatorId<TSchema>` helper
extracts their ids as a union, so APIs that take a group operator (like adding a nested group,
or changing how a group combines) only accept ids you declared.

```ts
// GroupOperatorId<typeof schema> is "and" | "or"
```
