The same query state as the [Basic](/docs/examples/basic) example, rendered as a flowing
**expression** of tokens instead of rows. It's built to be driven from the keyboard — click in
and try the shortcuts in the legend, or use your mouse.

<ModernPlayground />

## What it demonstrates

This is the standalone
[`examples/react/modern`](https://github.com/GRNRB/query-builder/tree/main/examples/react/modern)
app (an MUI / emotion Vite project) embedded in its own theme. It pushes the same hook far
beyond a form to show how much UI you can build on top of plain immutable trees:

| Feature | How it's done | Related |
| ------- | ------------- | ------- |
| **Undo / redo** | `useUndoableQuery` runs the hook in [controlled mode](/docs/guides/controlled-mode) and keeps `past` / `present` / `future` snapshots. | [Undo / redo](/docs/guides/undo-redo) |
| **Command palette** | A fuzzy picker for a token's field, operator, or combinator — every choice routes through `updateCondition` / `updateGroupOperator`. | [Schema](/docs/core/schema) |
| **Keyboard navigation** | `useQueryKeyboard` + `useFocusManager` map keys to mutations and move focus to the right token after each edit. | [API](/docs/frameworks/react/api) |
| **Arity-aware editing** | The value editor only opens for operators whose [arity](/docs/core/schema#operators-and-arity) takes a value; `is_set` has none. | [Arity](/docs/core/schema#operators-and-arity) |

The key takeaway: because a query is just an [immutable tree](/docs/core/query-tree), features
like undo/redo and focus restoration are *application* concerns layered on top — the library
stays a thin, predictable state engine underneath.

## Run it locally

```sh
git clone https://github.com/GRNRB/query-builder
cd query-builder
pnpm install
pnpm --filter './packages/*' run build   # build the libraries the example depends on
pnpm --filter modern dev                  # start the Vite app
```
