---
title: Immutability
description: Structural sharing, referential equality, and the safety guards.
---

Every tree operation is **immutable**: it returns a new tree rather than mutating the one
you passed in. Two properties follow from this, and both are useful.

## Structural sharing & referential equality

When a helper changes something, only the nodes on the path to the change are recreated —
untouched subtrees keep their original references. And when a helper changes *nothing*, it
returns the **exact same reference** it was given.

```ts
const next = appendToGroup(tree, "missing-group-id", node);
next === tree; // true — no such group, so nothing changed
```

This referential equality is a feature: you can cheaply detect "did anything change?" with
`===`, and React/memoization skip re-rendering subtrees that were structurally shared. The
helpers implement it by short-circuiting when a mapped child list is element-wise identical
to the original.

## Safety guards

A few operations refuse to do something destructive and return the tree unchanged instead:

- **`removeNode` won't remove the root.** The root group anchors the whole query, so
  removing it is a no-op (with a dev-mode warning).
- **`moveNode` won't orphan a subtree.** It refuses to move a node into itself or into one
  of its own descendants, and bails out if the target group doesn't exist — in every case
  returning the original tree rather than dropping nodes.

See the exact behavior of each in [Tree helpers](/docs/core/tree-helpers).

## Why no value is mutated

Because a query is plain, immutable data, it's trivial to:

- keep **history** (undo/redo) by holding past references,
- **serialize** the current tree at any moment (it's already JSON), and
- pass it through React state or any store without defensive copying.

<Callout>
Treat a `QueryNode` as frozen. Build changes through the [tree helpers](/docs/core/tree-helpers),
the [`reducer`](/docs/core/reducer), or the [React hook](/docs/react/use-query-builder) —
never by assigning into `node.children` or a condition's fields directly.
</Callout>
