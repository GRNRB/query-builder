title: Introduction

---

**query-builder** is a headless engine for query trees — nested conditions and groups,
operators that carry their own value shape, an immutable tree, and a typed reducer — with
**no markup of its own** and **no opinion about what your operators mean**. You declare the
vocabulary, you bring the components and the styling; it owns the hard parts.

## Why headless

A query builder is deceptively hard. Conditions nest into groups arbitrarily deep, each
operator takes a different _number_ of values (a range wants two, a set membership wants a
list, a presence check wants none), every edit has to produce a new immutable tree you can
store and diff, and TypeScript should stop you from pairing a field with an operator it
doesn't support.

Most libraries solve that by shipping a UI — and then you fight their markup, their styles,
and their component API to make it look like _your_ product. query-builder takes the
opposite stance: it hands you the **state and the operations**, and stays out of your render
tree entirely. The result is a filter builder that looks exactly the way you designed it,
with none of the tree-wrangling written by hand.

What you get:

- **A typed schema** — declare your fields and operators once; TypeScript enforces valid
  field/operator/value combinations everywhere downstream.
- **An immutable query tree** — plain JSON you can serialize, persist, diff, and time-travel.
- **Safe mutations** — add, insert, move, remove, and update via a reducer that guards
  against orphaned subtrees and resets incompatible values for you.
- **Zero UI assumptions** — no components, no CSS, no portals. Render it however you like.

## You define the vocabulary

query-builder ships **no operators and no semantics of its own**. There's no built-in
"and", no built-in "equals", no assumption that a group means a boolean combination. You
declare every operator your app uses — its id, its human label, and its **arity** (how many
values it takes: `none`, `one`, `two`, or `many`) — and the engine works entirely in terms
of _those_.

That keeps it paradigm-agnostic. A boolean filter with AND/OR groups is the obvious case,
but nothing stops you from modeling set algebra, full-text qualifiers, a SQL-`WHERE` dialect,
a feature-flag matcher, or a rules engine. The engine only manages the tree, the typing, and
the safe edits — what your operators _mean_ is yours to define, and yours to interpret when
you serialize or evaluate the result.

## Two ideas

The whole library rests on two pieces, and they're worth separating in your head:

- The **schema** is the _grammar_ — what queries are even allowed: which fields exist, which
  operators each field supports, and which operators may combine a group's children. It's
  static and carries no state. See [Schema](/docs/core/schema).
- The **query tree** is the _data_ — an actual query, a tree of groups and conditions. It's
  plain immutable JSON. See [Query tree](/docs/core/query-tree).

Everything else — the reducer and the tree helpers — is just a typed, safe way to turn one
valid tree into the next.

## Quick start

Install the core, declare a schema, build a tree, and advance it through the reducer:

```sh
npm install @grnrb/query-builder-core
```

```ts
import {
  defineSchema,
  normalizeTree,
  reducer,
} from "@grnrb/query-builder-core";

// The operators below are just what *this* app declared — pick whatever ids,
// labels, and arities your domain needs; the engine has no built-ins.
const schema = defineSchema({
  fields: [
    { id: "name", label: "Name", supportedOperators: ["eq", "contains"] },
    { id: "age", label: "Age", supportedOperators: ["eq", "gt", "lt"] },
  ],
  conditionOperators: [
    { id: "eq", arity: "one", label: "equals" },
    { id: "contains", arity: "one", label: "contains" },
    { id: "gt", arity: "one", label: "greater than" },
    { id: "lt", arity: "one", label: "less than" },
  ],
  groupOperators: [
    { id: "and", arity: "many", label: "AND" },
    { id: "or", arity: "many", label: "OR" },
  ],
});

// Build an initial tree — normalizeTree fills in any missing ids.
let query = normalizeTree({
  type: "group",
  operator: "and",
  children: [
    { type: "condition", field: "name", operator: "eq", value: "Alice" },
  ],
});

// Advance it with actions — each call returns a fresh, normalized tree.
query = reducer(query, {
  type: "APPEND_NODE",
  groupId: query.id,
  node: normalizeTree({
    type: "condition",
    field: "age",
    operator: "gt",
    value: 18,
  }),
});
```

From here the reducer accepts `APPEND_NODE`, `INSERT_NODE`, `MOVE_NODE`, `REMOVE_NODE`, and
the rest — each one returns a fresh, normalized tree. See the [Reducer &
actions](/docs/core/reducer) for the full set.
