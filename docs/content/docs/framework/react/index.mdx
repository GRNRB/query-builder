`@grnrb/react-query-builder` adds the **`useQueryBuilder`** hook over the core and
re-exports everything else (`defineSchema`, the tree helpers, all types) ŌĆö so a React app
imports only this one package.

## Install

```sh
npm install @grnrb/react-query-builder
# or: pnpm add @grnrb/react-query-builder
```

`react` (>= 18) is a peer dependency.

## Define a schema

The schema declares your fields, their operators, and how groups combine ŌĆö see
[Concepts](/docs/framework/react/concepts) for the full detail. Wrap it in `defineSchema`
so TypeScript infers the literal types.

```ts
import { defineSchema } from "@grnrb/react-query-builder";

const schema = defineSchema({
  fields: [
    { id: "name", label: "Name", supportedOperators: ["eq", "contains"] },
    { id: "age", label: "Age", supportedOperators: ["eq", "gt", "lt"] },
  ],
  conditionOperators: [
    { id: "eq", arity: "one", label: "equals" },
    { id: "contains", arity: "one", label: "contains" },
    { id: "gt", arity: "one", label: "greater than" },
    { id: "lt", arity: "one", label: "less than" },
  ],
  groupOperators: [
    { id: "and", arity: "many", label: "AND" },
    { id: "or", arity: "many", label: "OR" },
  ],
});
```

## Build a query

`useQueryBuilder` takes the schema and an `initialQuery` (ids optional ŌĆö the hook fills them
in via [`normalizeTree`](/docs/framework/react/reference#normalizetree)) and returns the
current `query` plus mutation helpers.

```tsx
import { useQueryBuilder } from "@grnrb/react-query-builder";

function Filters() {
  const qb = useQueryBuilder(schema, {
    initialQuery: {
      type: "group",
      operator: "and",
      children: [
        { type: "condition", field: "name", operator: "eq", value: "Alice" },
      ],
    },
  });

  return <pre>{JSON.stringify(qb.query, null, 2)}</pre>;
}
```

`qb.query` is the normalized [query tree](/docs/framework/react/concepts#the-query-tree); the
helpers are **stable** (memoized) callbacks that **normalize** your inputs for you.

## Mutate the tree

Call a helper with the id of the node you're targeting. Use `qb.query.id` (the root group's
id) to target the top level.

```ts
// Add
qb.addCondition(groupId, {
  type: "condition",
  field: "age",
  operator: "gt",
  value: 30,
});
qb.addGroup(groupId, "or", [
  { type: "condition", field: "status", operator: "eq", value: "active" },
]);

// Insert at an index
qb.insertCondition(groupId, 0, {
  type: "condition",
  field: "name",
  operator: "eq",
  value: "",
});
qb.insertGroup(groupId, 0, "and");

// Remove & move
qb.removeNode(nodeId); // removing the root is a no-op
qb.moveNode(nodeId, targetGroupId, index); // index optional ŌåÆ appends

// Update
qb.updateCondition(nodeId, { value: "Bob" }); // simple value change
qb.updateCondition(nodeId, { field: "age" }); // operator/value reset if incompatible
qb.updateGroupOperator(groupId, "or"); // toggle a group's operator
```

Each helper dispatches one [reducer action](/docs/framework/react/reference#reducer) under the hood:

| Helper                            | Action                  |
| --------------------------------- | ----------------------- |
| `addCondition` / `addGroup`       | `APPEND_NODE`           |
| `insertCondition` / `insertGroup` | `INSERT_NODE`           |
| `removeNode`                      | `REMOVE_NODE`           |
| `moveNode`                        | `MOVE_NODE`             |
| `updateCondition`                 | `UPDATE_NODE`           |
| `updateGroupOperator`             | `UPDATE_GROUP_OPERATOR` |

<Callout title="Field changes reset the operator/value">
  `updateCondition` runs
  [`buildConditionPatch`](/docs/framework/react/reference#buildconditionpatch)
  for you: if the new `field` doesn't support the current operator, the operator
  resets to the new field's default and the value clears. `moveNode` carries its
  [guards](/docs/framework/react/reference#tree-helpers) (it won't orphan a
  subtree). For drag-to-reorder, mind the index adjustment in [Drag and
  drop](#drag-and-drop) below.
</Callout>

## Controlled vs uncontrolled

Same return value and helpers either way ŌĆö only **who owns the state** differs.

**Uncontrolled** (above) ŌĆö pass `initialQuery`; the hook holds state in its own `useReducer`.
Simplest for self-contained builders.

**Controlled** ŌĆö pass `query` + `onChange`; you own the state and the hook calls `onChange`
with the next tree after every mutation. Use it when the query must live in a parent store,
be validated, debounced, or persisted.

```tsx
const [query, setQuery] = useState<QueryNode<typeof schema>>(() => ({
  id: "root",
  type: "group",
  operator: "and",
  children: [],
}));

const qb = useQueryBuilder(schema, { query, onChange: setQuery });
```

<Callout title="In controlled mode, seed the root id yourself">
  Because you create the initial `query` object directly (not via
  `initialQuery`), give the root a stable `id` (e.g. `"root"`). New nodes added
  through the helpers are still normalized for you.
</Callout>

## Custom id generation

Every node needs a unique `id`. By default the library uses `defaultGenerateId`
(`crypto.randomUUID()`). Pass `generateId` to the hook to override it:

```ts
let counter = 0;
const qb = useQueryBuilder(schema, {
  initialQuery,
  generateId: () => `node-${++counter}`,
});
```

Override it for **deterministic tests** (predictable snapshots), to avoid **SSR/hydration**
mismatches (`crypto.randomUUID()` is non-deterministic ŌĆö use a deterministic generator or
normalize once and pass the result down), or for **shorter/prefixed** ids.

<Callout type="warn">
  Whatever generator you use must return **unique** strings within a tree ŌĆö ids
  are how every operation finds a node.
</Callout>

## TypeScript inference

[`defineSchema`](/docs/framework/react/reference#defineschema) uses a `const` type parameter,
so every literal is preserved. Capture the schema type with `typeof` and reuse it:

```ts
const schema = defineSchema({
  /* ... */
});
type S = typeof schema;
```

From `S`, the types flow automatically:

- **Operators per field** ŌĆö `QueryNode<S>` only allows a condition's `operator` to be one of
  its `field`'s `supportedOperators`.
- **Value shape per operator** ŌĆö driven by [arity](/docs/framework/react/concepts#operators-and-arity)
  (`none` forbids `value`, `one` requires one).
- **Group operators** ŌĆö `GroupOperatorId<S>` is the union of your group operator ids.

A handy alias types child components against the hook's return:

```ts
type QB = ReturnType<typeof useQueryBuilder<S>>;
```

`ValidateFieldDefaults` makes a `defaultOperator` that isn't in `supportedOperators` a
**compile error**, caught at `defineSchema`.

## Serializing queries

A query tree is plain, serializable data ŌĆö `JSON.stringify` it directly.

```ts
localStorage.setItem("filters", JSON.stringify(qb.query)); // or send to your API
```

A persisted tree **already has ids**, so parse and use it as-is (as `initialQuery`). For a
**template authored without ids**, normalize it on load ŌĆö `normalizeTree` only fills
_missing_ ids, so it's safe either way:

```ts
import { normalizeTree } from "@grnrb/react-query-builder";

const tree = normalizeTree(templateWithoutIds);
```

Translating a tree into a backend filter (SQL, Mongo, ElasticsearchŌĆ”) is a recursive walk ŌĆö
handle the two `type`s and read each operator's arity to interpret `value`:

```ts
function toPredicate(node: QueryNode<typeof schema>): unknown {
  if (node.type === "group")
    return { [node.operator]: node.children.map(toPredicate) };
  return { field: node.field, op: node.operator, value: (node as any).value };
}
```

<Callout>
  Keep your **schema** as the source of truth on both ends, and validate an
  incoming tree against it before trusting persisted queries that may predate a
  schema change.
</Callout>

## Drag and drop

Reordering maps onto [`moveNode`](/docs/framework/react/reference#tree-helpers): on drop, call
`qb.moveNode(draggedId, targetGroupId, index)`. There is **one subtlety** ŌĆö `moveNode` removes
the dragged node first, then inserts. So when you drag a node **downward within the same
group**, the removal shifts later indices down by one; subtract one in that case.

```ts
// Find a node's parent group and its index within that group.
function findParentAndIndex(node, childId) {
  if (node.type !== "group") return null;
  for (let i = 0; i < node.children.length; i++) {
    if (node.children[i].id === childId) return { parentId: node.id, index: i };
    const found = findParentAndIndex(node.children[i], childId);
    if (found) return found;
  }
  return null;
}

function onDrop(query, draggingId, groupId, dropIndex) {
  const source = findParentAndIndex(query, draggingId);
  const insertIndex =
    source && source.parentId === groupId && source.index < dropIndex
      ? dropIndex - 1
      : dropIndex;
  qb.moveNode(draggingId, groupId, insertIndex);
}
```

Because reordering goes through `moveNode`, the
[guards](/docs/framework/react/reference#tree-helpers) apply automatically ŌĆö you can't orphan
a subtree by dropping a group into its own child. This is exactly the logic the
[playground](/docs/framework/react/playground) uses.
