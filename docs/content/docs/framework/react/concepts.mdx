Two ideas underpin the whole library: the **schema** (the grammar — what queries are
*allowed*) and the **query tree** (the data — an actual query). They're framework-agnostic;
in React you build and change the tree through the
[`useQueryBuilder`](/docs/framework/react) hook.

## Schema

The **schema** is a static description of *what's allowed*: which fields exist, which
operators each field supports, and how groups combine. It carries no state — it's the
grammar your queries are written in — and because it's defined with
[`defineSchema`](/docs/framework/react/reference#defineschema), TypeScript infers its literal
types and drives the typing of everything else.

```ts
import { defineSchema } from "@grnrb/react-query-builder";

const schema = defineSchema({
  fields: [
    { id: "name", label: "Name", supportedOperators: ["eq", "contains"] },
    { id: "age", label: "Age", supportedOperators: ["eq", "gt", "lt"] },
  ],
  conditionOperators: [
    { id: "eq", arity: "one", label: "equals" },
    { id: "contains", arity: "one", label: "contains" },
    { id: "gt", arity: "one", label: "greater than" },
    { id: "lt", arity: "one", label: "less than" },
  ],
  groupOperators: [
    { id: "and", arity: "many", label: "AND" },
    { id: "or", arity: "many", label: "OR" },
  ],
});
```

Only `fields` and `conditionOperators` are required; `groupOperators` is optional (but
you'll almost always want it).

### Fields

Each field declares an `id`, a human `label`, and the operator ids it `supportedOperators`.
An optional `defaultOperator` (which must be one of `supportedOperators`) is used when a
condition switches to this field.

## Operators and arity

Condition operators and group operators share one shape. The key field is **`arity`** — how
many values an operator takes — which determines the `value` shape of a condition using it.

<AutoTypeTable path="../packages/query-builder-core/src/type.ts" name="OperatorDefinition" />

```ts
type Arity = "none" | "one" | "two" | "many";
```

| Arity    | Example operator       | `value` shape              |
| -------- | ---------------------- | -------------------------- |
| `none`   | `is_set`, `is_empty`   | *(omitted / `undefined`)*  |
| `one`    | `eq`, `contains`, `gt` | a single value `TValue`    |
| `two`    | `between`              | a tuple `[TValue, TValue]` |
| `many`   | `in`, `not_in`         | an array `TValue[]`        |

A `none`-arity operator has **no value at all**:

```ts
// Active field whose only operator is `is_set` (arity: "none") — note: no `value` key.
{ type: "condition", field: "active", operator: "is_set" }
```

This is why, in the [playground](/docs/framework/react/playground), the value input is
rendered from the operator's arity: `one` shows a text box, `none` shows *(no value)*.

<Callout title="Value shape is enforced">
Because the schema knows each operator's arity, TypeScript enforces the right `value` per
condition — a `none` operator forbids a `value`, a `one` operator requires one. See
[TypeScript inference](/docs/framework/react#typescript-inference).
</Callout>

## Group operators

The valid group operators are whatever you put in `groupOperators` (typically `and` / `or`,
with `arity: "many"`). They combine a group's children. The `GroupOperatorId<TSchema>`
helper extracts their ids as a union, so APIs that take a group operator (like adding a
nested group, or changing how a group combines) only accept ids you declared.

```ts
// GroupOperatorId<typeof schema> is "and" | "or"
```

## The query tree

A query is plain JSON — a **tree** whose root is always a `group`. Each child is either a
`condition` (a leaf) or another `group` (so you can nest `and` / `or` arbitrarily deep).
There are no classes and no methods on the nodes; a query is just a serializable value you
can store, send over the wire, and diff. In React you change it through the
[`useQueryBuilder`](/docs/framework/react) hook.

```ts
{
  type: "group",
  operator: "and",
  children: [
    { type: "condition", field: "name", operator: "eq", value: "Alice" },
    {
      type: "group",
      operator: "or",
      children: [
        { type: "condition", field: "age", operator: "gt", value: 18 },
      ],
    },
  ],
}
```

### Condition nodes

A `ConditionNode` is a leaf: a single filter as `field` + `operator` + (usually) `value`.
The value's shape depends on the operator's [arity](#operators-and-arity).

```ts
{ id: "c1", type: "condition", field: "name", operator: "eq", value: "Alice" }
```

### Group nodes

A `GroupNode` combines other nodes with a logical `operator`. Its `children` are themselves
conditions or groups.

```ts
{ id: "g1", type: "group", operator: "and", children: [ /* ConditionNode | GroupNode */ ] }
```

### Every node has an id

Both node types carry a unique string `id`. Ids are how every operation targets a node — to
move, remove, or update it, you pass its id.

### Input variants — ids are optional

When you *build* a tree (an initial query, or a node you add), you usually don't invent ids
by hand. The `*Input` types make `id` optional: `ConditionNodeInput`, `GroupNodeInput`, and
the union `QueryNodeInput`. Pass an input, and
[`normalizeTree`](/docs/framework/react/reference#normalizetree) (called for you by the hook)
fills in any missing ids.

```ts
// No ids needed — normalizeTree / the hook assigns them.
{
  type: "group",
  operator: "and",
  children: [{ type: "condition", field: "age", operator: "gt", value: 18 }],
}
```

## Immutability

Every operation is **immutable**: it returns a *new* tree rather than mutating the one you
passed. Two properties follow, and both are useful.

**Structural sharing & referential equality.** When something changes, only the nodes on the
path to the change are recreated — untouched subtrees keep their references. When nothing
changes, the helper returns the **exact same reference**:

```ts
const next = appendToGroup(tree, "missing-group-id", node);
next === tree; // true — no such group, so nothing changed
```

You can detect "did anything change?" with `===`, and React skips re-rendering structurally
shared subtrees. `children` is declared `readonly` as a reminder to transform, not push.

**Safety guards.** A few operations refuse to do something destructive and return the tree
unchanged instead:

- **Removing the root is a no-op** — the root anchors the query (dev-mode warning).
- **Moving a node into itself or a descendant is rejected** — which would orphan the moved
  subtree (so is moving to a target that doesn't exist).

Because a query is plain immutable data, it's trivial to keep history (undo/redo), serialize
it ([it's already JSON](/docs/framework/react#serializing-queries)), and pass it through any
store without defensive copying.

<Callout>
Treat a `QueryNode` as frozen. Change it through the [hook helpers](/docs/framework/react#mutate-the-tree)
— never by assigning into `node.children` or a condition's fields directly.
</Callout>
