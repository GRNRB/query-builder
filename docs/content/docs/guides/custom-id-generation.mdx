---
title: Custom id generation
description: Replace the default UUID generator for tests, SSR, or shorter ids.
---

Every node needs a unique `id`. By default the library uses
[`defaultGenerateId`](/docs/core/tree-helpers#defaultgenerateid), which calls
`crypto.randomUUID()`. You can supply your own generator wherever a `generateId` parameter
appears.

## With the hook

```ts
let counter = 0;
const qb = useQueryBuilder(schema, {
  initialQuery,
  generateId: () => `node-${++counter}`,
});
```

The hook threads your `generateId` through `normalizeTree`, so every node it creates —
including the ones added via `addCondition` / `addGroup` — uses it.

## With the core helpers

```ts
import { normalizeTree } from "@grnrb/query-builder-core";

let n = 0;
const tree = normalizeTree(input, () => `n${++n}`);
```

## Why override it

- **Deterministic tests.** A counter makes ids predictable, so snapshots and assertions are
  stable.
- **SSR / hydration.** `crypto.randomUUID()` is non-deterministic, so a tree normalized on
  the server won't match one normalized on the client. Use a deterministic generator (or
  normalize once and pass the result down) to avoid hydration mismatches.
- **Shorter or prefixed ids.** Sometimes you want readable ids (`cond-1`) or a namespace.

<Callout type="warn" title="Keep ids unique">
Whatever generator you use, it must return **unique** strings within a tree — ids are how
every helper finds a node. A counter that resets, or a generator that can collide, will
make `findNode`/`moveNode`/`removeNode` target the wrong node.
</Callout>
